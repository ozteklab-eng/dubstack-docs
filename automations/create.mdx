Custom automations let you define a docs maintenance workflow for your team's specific process.

Use them when the built-in automations are close, but your trigger, prompt, context, or notification rules need to be more specific.

## Create a custom automation

1. Open the dashboard.
2. Select the deployment you want to configure.
3. Go to **Automate > Automations**.
4. Choose **Create a custom automation**.
5. Add a name and summary.
6. Select the trigger.
7. Add context repositories or integrations if needed.
8. Write clear instructions.
9. Choose the update mode.
10. Save the automation.

## Choose a trigger

| Trigger | When to use it |
| --- | --- |
| Content update | Run when docs content changes. |
| Code change | Run when selected source code changes merge. |
| Custom schedule | Run on a recurring schedule. |

Choose the narrowest trigger that fits the workflow. Broad triggers can create noisy updates.

## Add context repositories

Context repositories are cloned into the automation environment so the agent can read code while drafting docs.

They do not automatically trigger the automation unless you also choose a code-change trigger and the repository is selected as a trigger source.

## Add integration context

Connected integrations can provide bounded context for a run.

Examples:

- Use Slack context when the source discussion happened in a channel or thread.
- Use Jira context when issues describe a feature, bug, or release scope.

Only enable integrations that are relevant to the automation. This keeps prompts smaller and updates easier to review.

## Write effective instructions

Good instructions are specific:

```text
Draft a short troubleshooting page when the linked issue describes a recurring setup failure.
Use the existing troubleshooting page structure.
Keep code examples minimal.
Open a pull request for review.
```

Avoid instructions that are too broad:

```text
Improve the docs.
```

## Pick an update mode

| Mode | Behavior |
| --- | --- |
| Require review | Open a pull request and wait for review. |
| Automatically | Apply the update directly when the automation is allowed to write. |

Prefer review mode for new automations. Move to automatic updates only after the automation has produced predictable output across several runs.

## Validate a custom automation

After saving:

1. Run it manually if the automation is runnable.
2. Check the run history.
3. Review collected context sources.
4. Review the created pull request or direct update.
5. Adjust instructions if the output is too broad or too narrow.

<Callout type="warning">
  Automations can change documentation. Keep production workflows in review mode until the prompt, context, and output pattern are well understood.
</Callout>
