Tables are useful for compact comparisons, field lists, limits, and status matrices.

Use them when readers need to compare values across rows.

## Markdown table

```mdx
| Field | Type | Description |
| --- | --- | --- |
| `id` | `string` | Unique identifier. |
| `created_at` | `string` | ISO timestamp for creation. |
| `status` | `string` | Current state. |
```

Rendered result:

| Field | Type | Description |
| --- | --- | --- |
| `id` | `string` | Unique identifier. |
| `created_at` | `string` | ISO timestamp for creation. |
| `status` | `string` | Current state. |

## Comparison table

```mdx
| Option | Best for | Tradeoff |
| --- | --- | --- |
| Connect Git | Teams with an existing docs repo. | Requires provider authorization. |
| Hosted repository | Teams that want to start quickly. | Move to customer-owned Git later if needed. |
```

## Keep tables readable

- Use short column names.
- Keep cells concise.
- Move long explanations below the table.
- Avoid tables with too many columns on mobile.
- Use code formatting for literal values.

## API fields

For request and response schemas, use parameter or response field components when the content is more like a schema than a comparison:

```mdx
<ParamField path="limit" type="number">
  Maximum number of records to return.
</ParamField>
```

Use tables for overview comparisons, not deeply nested API schemas.
