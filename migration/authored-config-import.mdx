Authored config import is for repositories that already have documentation structure.

The import reads the existing docs configuration and maps it to `dubstack.json` where possible.

## Good candidates

Use authored config import when the source repository has:

- MDX or Markdown documentation pages
- a navigation/config file
- a predictable assets directory
- OpenAPI specs or API reference metadata
- existing route structure you want to preserve

## What gets mapped

| Source concept | dubstack target |
| --- | --- |
| Site name | `name` |
| Navigation groups | `navigation.tabs` and `navigation.groups` |
| Page routes | page slugs and MDX file paths |
| API specs | `openapi` references |
| Theme basics | `theme` settings when compatible |
| Redirects | `redirects` entries when available |

Not every source setting has a one-to-one equivalent. Review the imported config before publishing.

## Review checklist

After import:

- confirm every navigation item opens a page
- confirm renamed routes have redirects
- confirm OpenAPI operations render correctly
- confirm images and relative links resolve
- remove unsupported source-only settings
- commit the converted `dubstack.json`

## When not to use it

Do not use authored config import for repositories that only contain application code and no docs configuration. Use generated reconstruction for those repositories instead.
