Generated reconstruction creates a documentation preview for repositories that do not already have a complete docs structure.

It is useful for evaluating how a codebase could become a docs site, but it should be reviewed before becoming production documentation.

## When to use generated reconstruction

Use it when a repository has:

- source code but little or no documentation
- scattered Markdown files without navigation
- examples or API handlers that need explanation
- a need for an acquisition or migration preview

## Output

A reconstruction can produce:

- proposed navigation
- generated MDX pages
- overview and setup content
- API or module descriptions where signals exist
- a preview site for review

## Review required

Generated documentation must be reviewed by a human before production use.

Check for:

- incorrect assumptions
- missing prerequisites
- code paths described as public APIs when they are internal
- stale package names or commands
- incomplete security notes
- pages that should be removed or rewritten

## Promote to owned docs

After the generated preview is useful:

1. Claim or adopt the preview into a workspace.
2. Connect the appropriate repository or hosted source.
3. Review and edit generated pages.
4. Publish from the owned source.
5. Add redirects and a custom domain if needed.

<Callout type="warning">
  Generated reconstruction is not a replacement for product ownership. Treat it as a starting point that accelerates docs creation.
</Callout>
