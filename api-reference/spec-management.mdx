Use **API Specs** when you want to manage OpenAPI or AsyncAPI files from the dashboard instead of editing the spec reference manually.

The spec manager is deployment-scoped. It reads and writes through the connected docs source so published API pages stay reproducible.

## What you can manage

| Control | Behavior |
| --- | --- |
| Upload spec | Adds an OpenAPI or AsyncAPI JSON file to the deployment spec catalog. |
| Base URL | Sets the default origin used by the browser playground when the spec does not define a server URL. |
| Catalog visibility | Shows or hides the spec from dashboard catalog views without publishing every endpoint. |
| Generate assets | Creates endpoint MDX pages and reusable snippets from the selected OpenAPI spec. |
| Generate snippets | Regenerates reusable endpoint snippets without changing page visibility. |
| Refresh | Reloads the spec from its configured base URL when refresh is supported. |
| Manage endpoints | Publishes, hides, or excludes generated endpoint pages one operation at a time. |

## Upload a spec

1. Open the dashboard.
2. Go to **API Specs**.
3. Choose **Upload Spec**.
4. Upload a JSON OpenAPI or AsyncAPI file.
5. Confirm the name, display name, description, type, category, and base URL.
6. Choose whether the spec should appear in the catalog.
7. Save the spec.

For OpenAPI specs, include `servers[0].url` in the spec or provide **Base URL** in the form. The playground uses that value to build requests.

## Generate endpoint pages

After uploading an OpenAPI spec, use **Generate Assets** to create endpoint pages and supporting snippets.

Generated endpoint pages are not automatically made visible to readers. Use **Manage endpoints** to choose which operations should be published.

Endpoint states:

| State | Meaning |
| --- | --- |
| Published | The endpoint page is visible in the public API reference. |
| Hidden | The endpoint is known to the system but not shown in public navigation. |
| Excluded | The endpoint is intentionally kept out of generated docs. |
| Unpublished | Assets may exist, but the endpoint has not been published yet. |

## Keep the spec source clear

For Git-backed sites, the most predictable path is to keep the canonical spec in the docs repository and reference it from `dubstack.json`.

Use dashboard uploads when:

- a teammate needs to inspect a spec before committing a longer-term source change
- you need to generate endpoint pages from a known JSON spec
- you want to control endpoint visibility without editing every operation manually

Use repository changes when:

- the spec is generated by your API build pipeline
- the spec should be reviewed in pull requests
- the spec belongs beside versioned documentation content
