## How publishing works

A dubstack site has three layers:

| Layer | Purpose |
| --- | --- |
| Source | Your MDX files, assets, OpenAPI specs, and `dubstack.json` in Git. |
| Sync | The publish path that turns source content into versioned site artifacts. |
| Runtime | The public docs frontend on `dubstack.live` or your custom domain. |

```text
Git source
-> dubstack sync
-> published content artifacts
-> docs frontend
```

This keeps editing and publishing predictable. The dashboard can help you edit files, but the published site is still driven by synchronized source content.

## What to configure first

1. Create the site and choose a source option.
2. Confirm the live preview opens on the generated `dubstack.live` domain.
3. Update the homepage or quickstart page.
4. Review `dubstack.json` so navigation matches your intended structure.
5. Add an OpenAPI spec if you have API reference docs.
6. Invite teammates after the site has a working source and deploy path.

## Recommended first pages

Start with a small set of pages that answer real user questions:

- Overview or introduction
- Quickstart
- Authentication or setup guide
- Core concepts
- API reference introduction
- Troubleshooting

Add deeper sections after the first publish is working.

<Callout type="info">
  Keep each first change small. A focused first publish verifies the full source, sync, and runtime path before your team migrates larger docs sets.
</Callout>
