This quickstart takes you from an empty workspace to a published documentation site with real source control, navigation, and a live docs URL.

You can connect your own Git repository, or use the hosted repository flow when you want dubstack to create the starter source for you.

## What you will build

By the end, you will have:

- a docs site with a `dubstack.live` URL
- a source repository or hosted starter source
- a `dubstack.json` configuration file
- at least one edited MDX page
- a completed publish activity
- a live page you can open and verify

## Before you start

You need:

- a dubstack account
- a workspace or organization for the site
- a GitHub or GitLab account if you want to connect your own repository
- DNS access only if you plan to attach a custom domain later

## 1. Create a site

Open the dashboard and start the new site flow.

Choose the kind of docs you are building, then pick a source option.

| Source option | Use it when |
| --- | --- |
| Connect Git | You already have a docs repository, or your team wants the repository under its own Git provider account from day one. |
| Hosted repository | You want a working starter immediately and plan to edit from the dashboard first. |

After setup finishes, the dashboard opens on the site's overview page.

## 2. Review the starter source

The starter source includes MDX pages, assets, and `dubstack.json`.

A minimal source usually looks like this:

```text
.
├── dubstack.json
├── index.mdx
├── quickstart.mdx
├── api-reference/
│   └── openapi.json
└── assets/
    ├── logo-dark.svg
    └── logo-light.svg
```

The exact files depend on the starter profile and whether your site includes an API reference.

## 3. Check `dubstack.json`

`dubstack.json` controls navigation, theme, API reference setup, widgets, redirects, and footer behavior.

A small config can start like this:

```json
{
  "site": {
    "name": "Product Docs"
  },
  "navigation": {
    "tabs": [
      {
        "tab": "Guides",
        "default": true,
        "groups": [
          {
            "group": "Getting started",
            "pages": [
              { "slug": "index" },
              { "slug": "quickstart" }
            ]
          }
        ]
      }
    ]
  }
}
```

Keep the config committed with the docs source so every publish is reproducible.

## 4. Make your first edit

Open the editor from the dashboard.

A good first change is to edit `quickstart.mdx` or create a short page under a getting-started section.

Use the editor to:

1. update MDX content
2. insert supported components with slash commands
3. preview the rendered page
4. save the file back to the connected source

For example, add a callout to a page:

```mdx
<Callout type="info">
  This page is now managed through dubstack.
</Callout>
```

## 5. Publish

When you publish, dubstack follows the same production path used for normal updates.

```text
Git commit
-> sync worker
-> published content artifacts
-> docs frontend
```

Use the Activity panel on the dashboard to confirm the update completed.

If your source changed outside the normal webhook path, use Manual update from the overview page.

## 6. Verify the live site

After the publish finishes, check:

- the live site opens on the `dubstack.live` domain
- the edited page is visible after a hard refresh
- sidebar navigation matches `dubstack.json`
- search opens from the header
- the assistant opens and shows the reader notice
- no draft-only content is visible on the public site
