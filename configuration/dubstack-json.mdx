`dubstack.json` is the site configuration file in your docs repository.

Use it to control navigation, theme settings, OpenAPI references, redirects, contextual actions, and other published-site behavior.

## Minimal file

```json
{
  "site": {
    "name": "Acme Docs",
    "logo": {
      "light": "/assets/logo-light.svg",
      "dark": "/assets/logo-dark.svg"
    },
    "favicon": "/assets/favicon.svg"
  },
  "navigation": {
    "tabs": [
      {
        "tab": "Guides",
        "default": true,
        "groups": [
          {
            "group": "Getting started",
            "pages": ["index", "quickstart"]
          }
        ]
      }
    ]
  }
}
```

## Core fields

| Field | Purpose |
| --- | --- |
| `site` | Display name, tagline, logo, favicon, and other site-level branding. |
| `navigation` | Tabs, groups, pages, anchors, and OpenAPI-generated pages. |
| `theme` | Theme shell preset used for sizing, navigation, and page layout. |
| `palette` | Color-token preset layered onto the selected theme shell. |
| `colors` | Primary and accent colors used by active states, links, and highlights. |
| `styling` | Visual overrides such as code block behavior. |
| `redirects` | Source-to-destination redirects for renamed or moved pages. |
| `contextual` | Page actions such as copy Markdown, view Markdown, assistant, and MCP actions. |
| `api` | Default OpenAPI spec reference and playground display mode. |

## Site branding

Put public site branding under `site`.

```json
{
  "site": {
    "name": "Acme Docs",
    "tagline": "Developer documentation for Acme.",
    "logo": {
      "light": "/assets/logo-light.svg",
      "dark": "/assets/logo-dark.svg"
    },
    "favicon": "/assets/favicon.svg"
  }
}
```

Keep image paths repository-relative or root-relative so the publish pipeline can copy them to the published asset store.

## Source of truth

Keep `dubstack.json` committed with your docs. Dashboard configuration pages write back to the same source-backed configuration where supported.

A publish uses the configuration from the synced commit, not unsaved local state.

## Navigation references

Use page slugs without the `.mdx` extension:

```json
{
  "group": "Guides",
  "pages": [
    "index",
    "quickstart",
    "guides/authentication"
  ]
}
```

For a file at `guides/authentication.mdx`, the slug is `guides/authentication`.

## OpenAPI references

Reference a spec from a navigation group to generate endpoint pages:

```json
{
  "group": "API reference",
  "openapi": "api-reference/openapi.json"
}
```

List operations in `pages` when you only want selected endpoints visible:

```json
{
  "group": "API reference",
  "openapi": "api-reference/openapi.json",
  "pages": ["GET /v1/projects", "POST /v1/projects"]
}
```

## Add-ons configuration

Dashboard add-ons are stored under the `addons` key when the deployment supports writing those settings back to source.

```json
{
  "addons": {
    "feedback": {
      "thumbs_rating": true,
      "edit_suggestions": false,
      "raise_issues": false,
      "contextual_feedback": false,
      "code_snippet_feedback": false,
      "agent_feedback": false
    },
    "ci_checks": {
      "broken_links": "disabled",
      "grammar_linter": "disabled"
    },
    "previews": {
      "preview_deployments": true,
      "preview_deployment_authentication": false
    },
    "visibility": {
      "noindex": false
    }
  }
}
```

CI check values can be `disabled`, `advisory`, or `required`. Unsupported add-ons are forced back to a safe disabled state by the dashboard API.

## Validate changes

Before publishing configuration changes:

- Confirm every page slug maps to a file.
- Confirm OpenAPI paths and methods exist in the spec.
- Confirm redirects point to current pages.
- Commit the change before expecting it on the public site.

<Callout type="warning">
  Do not store API keys, passwords, or private tokens in `dubstack.json`. Use dashboard-managed secrets and integration settings for credentials.
</Callout>
